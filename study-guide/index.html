🔵 Consistent Hash Ring — Deep Dive

The core data structure is a ConcurrentSkipListMap<Long, HashableNode> — keys are virtual node hash positions, values are physical nodes. Skip list gives O(log n) ceiling lookup (finding next node clockwise).

Why Consistent Hashing?

With simple modulo hashing (key % N), adding/removing a node remaps ~all keys. Consistent hashing remaps only K/N keys on average — only the keys that "belonged" to the changed node move.

Problem: 4 nodes → 5 nodes: only 1/5 of keys migrate. Modulo: ~80% of keys would move.

Virtual Nodes (150 per physical)

Each physical node gets 150 virtual positions on the ring. Without vnodes, uneven distribution is guaranteed when nodes have similar hashes. Vnodes interleave all nodes evenly.

Total ring positions = 150 × N physical nodes. Each virtual node is hashed as "nodeId#vnode-0", "nodeId#vnode-1", etc.

💻 Core Implementation

@Component
public class ConsistentHashRing<T extends HashableNode> {
    private final ConcurrentSkipListMap<Long, T> ring = new ConcurrentSkipListMap<>();
    private final StampedLock lock = new StampedLock();  // optimistic reads
    private final int virtualNodes = 150;

    public void addNode(T node) {
        long stamp = lock.writeLock();
        try {
            for (int i = 0; i < virtualNodes; i++) {
                long hash = hashProvider.hash(node.nodeId() + "#vnode-" + i);
                ring.put(hash, node);
            }
        } finally { lock.unlockWrite(stamp); }
    }

    public T getNode(String key) {
        long stamp = lock.tryOptimisticRead();   // optimistic — no lock
        T result = lookupNode(key);
        if (!lock.validate(stamp)) {              // concurrent write happened
            stamp = lock.readLock();              // fall back to real read lock
            try { result = lookupNode(key); }
            finally { lock.unlockRead(stamp); }
        }
        return result;
    }

    private T lookupNode(String key) {
        long hash = hashProvider.hash(key);
        Map.Entry<Long, T> entry = ring.ceilingEntry(hash);  // walk clockwise
        return entry != null ? entry.getValue() : ring.firstEntry().getValue(); // wrap around
    }
}

📋 Migration Decision Logic

For each key in the local cache store, the service computes where the key SHOULD live in the new ring and decides what action to take:

for each key in localCacheStore:
    newReplicaSet = newRing.getNodes(key, replicationFactor)  // N=3
    isSelfInNewReplicas = newReplicaSet.contains(selfNodeId)

    if !isSelfInNewReplicas:
        action = EVICT        // I no longer own this key → push to primary, delete

    else if newNodesJoined:
        action = SEED         // New nodes need my data → push to new nodes

    else if nodesRemoved:
        action = RE_SEED      // Surviving replicas need to re-sync each other

    else:
        action = SKIP         // No change for this key

📡 Gossip Protocol

🔀 MembershipMerger — Conflict Resolution

When two nodes exchange state for the same peer, they must deterministically resolve conflicts:

// Priority order (highest wins):
// 1. Higher incarnation number always wins
// 2. Same incarnation → higher severity wins (DEAD > SUSPECT > ALIVE)
// 3. Same incarnation + same status → newer heartbeat wins

boolean shouldAccept(NodeInfo existing, NodeInfo incoming) {
    if (incoming.incarnation > existing.incarnation) return true;
    if (incoming.incarnation == existing.incarnation) {
        if (severity(incoming.state) > severity(existing.state)) return true;
        if (severity(incoming.state) == severity(existing.state))
            return incoming.heartbeat > existing.heartbeat;
    }
    return false;
}
Incarnation vector prevents split-brain resurrection: a node marked DEAD with incarnation=5 cannot be brought back to ALIVE with a stale message from incarnation=3.

🩺 Failure Detection (SWIM-style)

2
If no ACK → Indirect PING_REQAsks K random peers to each ping the suspect. "Can you reach node X?" If any helper gets ACK, node is still alive.
3

PeerSelector — Lock-free Round-Robin

private final AtomicInteger cursor = new AtomicInteger(0);
private volatile List<String> probeRing;  // shuffled peers

public String nextTarget() {
    int idx = cursor.getAndIncrement() % probeRing.size();
    return probeRing.get(idx);
}

public List<String> selectHelpers(String target, int k) {
    // pick k+1 random peers, filter out target
    return probeRing.stream()
        .filter(p -> !p.equals(target))
        .limit(k).collect(toList());
}
Uses java.util.Random (not SecureRandom) — no crypto needed, ~100x faster for gossip sampling.

💻 QuorumService — Reactive Code

public Mono<WalWriteMetadata> quorumWriteWithMetadata(String key, byte[] data, ...) {
    List<CacheNode> replicas = cacheRouter.routeToReplicas(key, replicationFactor);
    long version = versionCounter.incrementAndGet();

    List<Mono<NodeResponse>> writes = replicas.stream()
        .map(node -> httpClient.forwardPutRequest(node, key, data, version, expiresAt)
            .map(r -> new NodeResponse(node, true))
            .onErrorReturn(new NodeResponse(node, false)))
        .collect(toList());

    return Flux.merge(writes)
        .collectList()
        .flatMap(responses -> {
            long successes = responses.stream().filter(NodeResponse::success).count();
            if (successes < writeQuorum)
                return Mono.error(new QuorumNotMetException(...));
            return Mono.just(new WalWriteMetadata(version, successNodes, failedNodes));
        });
}

🔵 Consistent Hashing

What is consistent hashing and why does EdgeFabric use it? ▼
Consistent hashing maps both keys and nodes to positions on a virtual ring using the same hash function. To find the owner of a key, you hash the key and walk clockwise to the next node. The key advantage: when a node is added or removed, only ~K/N keys need to migrate (K = total keys, N = node count) instead of remapping ~all keys as with modulo hashing. EdgeFabric uses it so that adding a 4th cache node only migrates ~25% of keys rather than re-routing everything.
What are virtual nodes and why does EdgeFabric use 150 per physical node? ▼
Without virtual nodes, physical nodes hash to positions on the ring that may be unevenly spaced — one node might own 60% of the ring, another 10%. Virtual nodes create 150 hash positions per physical node (hashed as "nodeId#vnode-0" through "nodeId#vnode-149"). These interleave across the ring so each physical node ends up owning approximately equal share. With 150 vnodes per node, standard deviation of load distribution drops to ~5% vs ~30% without vnodes. 150 is a tuned balance — too few means poor distribution, too many wastes memory on the skip list.
Why ConcurrentSkipListMap and not a TreeMap or HashMap? ▼
  • TreeMap is not thread-safe — requires external synchronization, which would turn the StampedLock optimization pointless.
  • HashMap has O(1) lookup but cannot do ceiling lookup (find next node clockwise). ConcurrentSkipListMap supports ceilingEntry(hash) natively in O(log n).
  • ConcurrentSkipListMap is thread-safe, ordered, and supports the ceiling/floor operations needed for ring traversal. It also has lower contention than a synchronized TreeMap under concurrent reads.
Explain the StampedLock optimistic read pattern used in ConsistentHashRing ▼
StampedLock has three modes: write lock, read lock, and optimistic read. Optimistic read acquires no lock at all — it just saves a stamp. After the read, it calls validate(stamp). If a write happened between the stamp and validate, validate returns false and we fall back to a real read lock. In EdgeFabric, node topology changes (addNode/removeNode) are rare, so 99.9% of getNode calls succeed with optimistic read — zero lock overhead. Only on the rare concurrent topology change do we pay for a real read lock. This is the lowest-latency read strategy possible short of lock-free.
How does the ring handle the "wrap-around" case? ▼
ring.ceilingEntry(hash) returns the next node at or above the hash position. If the hash is greater than all ring positions (end of ring), ceilingEntry returns null. In that case, EdgeFabric uses ring.firstEntry().getValue() — the node at the beginning of the ring. This implements the circular "wrap-around" semantic of the ring.
Why XXHash over MurmurHash3? ▼
XXHash (net.openhft:zero-allocation-hashing) produces zero heap objects per call — it operates entirely on stack. MurmurHash3 via Guava allocates some temporary objects. At high throughput (millions of lookups/sec), GC pressure from hash calls can cause latency spikes. XXHash is also ~10x faster in throughput benchmarks. Both have excellent distribution quality. MurmurHash3 is kept as a fallback (configurable via ring.hash-algorithm in application.yml).

🔀 Key Migration

Walk me through what happens when a new cache node joins the cluster ▼
  • New node starts, reads seed peers from config/DNS
  • Sends POST /cluster/join to seeds — receives full membership digest
  • Starts gossip loop; topology change propagates to all nodes in O(log N) rounds
  • Each node's gossip protocol detects the new member → fires TopologyChangedEvent
  • KeyMigrationService receives event, debounces (batches rapid changes)
  • Rebuilds ring with new node → scans all local keys
  • For each key: if new node is now in replica set and self is elected seeder → SEED new node with that key's data
  • ClusterSyncService on the LB also re-syncs (every 5s) → adds new node to CacheRouter → future requests route to new node
Why is there a 15-second delayed deletion after EVICT? ▼
When a node decides to EVICT a key (push to new owner and delete locally), there's a race condition: the load balancer still has the old ring and might route a read request to this node for that key. If we deleted immediately, the LB would get a 404 even though the key was just migrated. The 15-second delay gives the LB's ClusterSyncService (which runs every 5s) time to discover the new ring topology and stop routing that key to the old node. 15s = 3× the LB sync interval, providing a safety buffer.
What is the thundering herd problem in SEED and how does seeder election prevent it? ▼
When a new node joins, multiple existing replicas all have the same key. Without coordination, all of them would simultaneously try to SEED the new node with the same key — wasting bandwidth and potentially causing write conflicts. EdgeFabric uses seeder election: the first non-new node in the new replica set is elected as the single seeder. All other replicas with that key detect they're not the elected seeder and skip SEED for that key. Result: exactly one node seeds each key to the new node, eliminating the thundering herd.
How does migration handle a topology change that happens DURING migration? ▼
MigrationWorker has a cancel flag (volatile boolean or AtomicBoolean). When KeyMigrationService receives a new TopologyChangedEvent while migration is in progress, it sets the cancel flag on the running worker. The worker checks this flag in its per-key loop and aborts when set. KeyMigrationService then computes a fresh migration plan from the current (latest) topology and starts a new MigrationWorker. This ensures migration is always based on the most current ring state, not a stale snapshot.
How do you avoid losing data during key migration? ▼
  • Push before delete: EVICT path always PUTs key to new owner before scheduling deletion. Data exists on both nodes during the transition window.
  • Delayed delete (15s): Key stays on old node for 15s after push, ensuring LB routes requests to new owner first.
  • Retry with backoff: If push fails, worker retries with exponential backoff (100ms base, 10s max). Only deletes after confirmed push.
  • WAL: All writes are logged to WAL. Anti-entropy process can replay WAL to recover any key that was lost during migration failure.
  • Quorum reads: R=2 means a read needs 2 replicas to agree. Even if one replica is mid-migration, the other two can serve the request.
What's the rate limit on migration and why? ▼
Default: 500 keys/second. Without rate limiting, migration would consume all network bandwidth and CPU, starving normal cache operations. The rate limiter uses a busy-wait spin loop (not Thread.sleep) because 1/500 keys/sec = 2ms intervals and Thread.sleep has ~1ms jitter — too coarse for this granularity. The spin loop checks System.nanoTime() and yields until the next emission time, giving sub-millisecond precision. The rate is configurable — in practice you'd tune it based on observed cluster load.

📡 Gossip & Failure Detection

Why gossip instead of a centralized registry? ▼
A centralized registry (like ZooKeeper or Consul) is a single point of failure and a scalability bottleneck. Gossip is decentralized: every node maintains its own membership view and disseminates updates peer-to-peer. Advantages: no SPOF, scales to large clusters (convergence is O(log N)), network partition tolerant (each partition eventually reconverges independently). Disadvantage: eventual consistency — there's a brief window where nodes have inconsistent views. For EdgeFabric's use case (cache routing), brief inconsistency is acceptable; quorum operations handle the correctness guarantee.
Why mark nodes SUSPECT before DEAD? Why not immediately DEAD? ▼
Network hiccups, GC pauses, and load spikes can cause temporary unresponsiveness. Immediately marking a node DEAD would trigger unnecessary key migration and gossip churn. The SUSPECT state is a grace period (10 seconds default): the node gets indirect PING_REQ probes from K other peers. If any helper reaches the suspect node, it's still alive — just temporarily unreachable from the detector. This SWIM-style indirect probing dramatically reduces false positives. Only after the suspect timeout with no ACK from any path is the node promoted to DEAD.
How does the incarnation vector prevent a dead node from being resurrected? ▼
Each node maintains an incarnation number that starts at 0 and increments when the node restarts or refutes a SUSPECT. The MembershipMerger only accepts state updates with incarnation ≥ existing. So: if node A is marked DEAD with incarnation=5, a stale gossip message saying A is ALIVE with incarnation=3 is rejected. Only A itself can override this by bumping its incarnation to 6+ via refutation — which it can only do if it's actually alive and able to send messages.
How fast does gossip converge and why? ▼
With fanout=2 and interval=1s, convergence is O(log₂ N) rounds. For a 10-node cluster: log₂(10) ≈ 3.3 → ~4 seconds for all nodes to see a topology change. This is because gossip spreads information like an epidemic: after each round the number of informed nodes doubles (fanout=2). The logarithmic convergence makes gossip highly scalable — doubling the cluster size adds only one more round. For EdgeFabric's expected cluster sizes (4–16 nodes), convergence is under 5 seconds.

⚖️ Quorum & Consistency

Why W=2, R=2, N=3? What guarantee does this provide? ▼
The quorum guarantee: W + R > N. With 2 + 2 > 3, the write quorum and read quorum always overlap by at least one node. This means every read will include at least one node that received the latest write. In practice: a write succeeds when 2 of 3 replicas ack. A read succeeds when 2 of 3 respond. At least one node is in both sets → it has the latest version. EdgeFabric then uses the highest version among read responses as the winner. Setting W=R=2 with N=3 allows one node failure while maintaining both read and write availability.
How does version numbering work in EdgeFabric? ▼
The load balancer's QuorumService owns version assignment via an AtomicLong seeded from System.currentTimeMillis() at startup. Each write call does versionCounter.incrementAndGet(). The version is passed to all N replica nodes as the X-Quorum-Version header. Cache nodes store the version alongside the data. On read, replicas return their stored version; the LB selects the response with the highest version as the winner. Keeping version assignment centralized (at LB) ensures global monotonic ordering across all writes.
What happens if only 1 of 3 cache nodes is available? ▼
Both read and write quorums require 2 responses. With only 1 node available:
  • Write: QuorumService waits for W=2 acks, gets only 1 → throws QuorumNotMetException → LB returns 503 to client
  • Read: QuorumService waits for R=2 responses, gets only 1 → same exception → 503
This is a CP trade-off: EdgeFabric sacrifices availability (returns error) to maintain consistency (refuses to serve potentially stale data). The WAL captures the failed write so anti-entropy can retry when the downed node recovers.
What is read repair and when is it triggered? ▼
Read repair is triggered when a quorum read collects R=2 successful responses but they have different versions — indicating a stale replica. The winner (highest version) is returned to the client. Asynchronously (non-blocking), ReadRepairService sends PUT requests to all stale replicas with the winner's data and version. This repairs replicas that missed a write (e.g., were down when the write was issued). It's a background self-healing mechanism that improves consistency over time without adding latency to the read path.

⚙️ System Design & Architecture

How does multi-tenancy work in EdgeFabric? ▼
Multi-tenancy is implemented via key namespacing. Every cache key is prefixed with the tenant identifier: tenant:key. The consistent hash ring routes on the full prefixed key, so different tenants' keys naturally distribute across the same nodes. There's no per-tenant ring — the prefix ensures key isolation at the data level. The load balancer validates key format against a regex pattern (^[a-zA-Z0-9:_-]+$) and extracts tenant from the URL path. This approach is simple and scalable — adding a new tenant requires no infrastructure changes.
How does service discovery work — Cloud Map vs Docker DNS? ▼
EdgeFabric supports both:
  • Docker Compose (dev): All cache nodes share the DNS alias cache-nodes.cache-cluster.internal. Docker's internal DNS round-robins between all containers with that alias. LB resolves this DNS to get all node IPs.
  • AWS Production: Cache nodes call AWS ServiceDiscovery API on startup (@PostConstruct) to register their private IP in Cloud Map service srv-6lnd44knosnojplq. On shutdown (@PreDestroy), they deregister. The LB resolves the Cloud Map DNS name cache-nodes.cache-cluster.internal which returns all registered IPs. Cloud Map is enabled via cloudmap.enabled=true.
ClusterSyncService tries peer gossip first, falls back to DNS if peers are unavailable.
Why use SSM for deployment instead of ECS or Kubernetes? ▼
SSM (Systems Manager) is simpler and cheaper for this scale. ECS/Kubernetes add significant operational complexity (control plane, task definitions, service meshes). With SSM: Jenkins discovers EC2 instances by tag (Role=hermes-cache-node), sends shell commands via SSM RunCommand to pull the latest Docker Hub image and restart the container — no SSH keys needed, fully automated, IAM-controlled. The tradeoff is less built-in autoscaling and orchestration, but EdgeFabric's gossip-based membership and manual ring scaling are sufficient for current requirements.
What is the WAL and how does it support anti-entropy? ▼
The Write-Ahead Log records every PUT operation with its key, data, version, and — critically — which nodes succeeded and which failed (successNodes / failedNodes in WalWriteMetadata). When a write achieves W=2 acks but the 3rd node was unavailable, the WAL records the 3rd node as failed. An anti-entropy background process periodically reads the WAL and retries writes specifically to the failedNodes — targeted repair rather than a full key scan. This ensures eventual consistency: every node that should have a key eventually gets it, even across transient failures.
What is structured logging and why does EdgeFabric use it? ▼
Every log line is output as a JSON object with fields: timestamp, level, service, traceId, nodeId, operation, duration, statusCode, result, key. This is "structured logging" — machine-readable logs. Benefits:
  • Loki/Elasticsearch can index and query any field (e.g., "show all PUT ops with duration > 100ms for tenant=acme-corp")
  • Agentic ops can parse logs programmatically without regex
  • traceId links logs across LB and all cache nodes for a single request (distributed tracing)
Implementation: Logstash Logback Encoder writes JSON, MDC fields are included automatically, async appenders prevent I/O blocking the request thread.
How does grpc transport work for GET operations? ▼
GET transport is configurable via cluster.get.transport: grpc (vs http). In Docker Compose, gRPC is used for cache node GET calls on port 9091. gRPC benefits:
  • HTTP/2 multiplexing — multiple concurrent requests over one connection, reducing connection overhead
  • Binary protobuf encoding — smaller payloads than JSON
  • Built-in streaming support
For quorum reads (3 parallel GETs), multiplexing means all 3 requests share one connection per node vs 3 separate HTTP/1.1 connections. This reduces latency and connection pool pressure, especially important for the read-heavy cache workload.

🔒 JVM & Performance

What JVM tuning does EdgeFabric use and why? ▼
From the load balancer Dockerfile:
  • -XX:+UseG1GC — G1 GC for low pause times with large heaps
  • -XX:MaxGCPauseMillis=20 — target max 20ms GC pauses (P99 latency SLA)
  • -Xmx512m — 512MB heap limit (prevents OOM on the EC2 instance)
  • -XX:+UseStringDeduplication — G1 deduplicates identical strings (cache keys, tenant names are highly repeated) → reduces heap by ~10-20%
  • -XX:+HeapDumpOnOutOfMemoryError — auto heap dump on OOM for post-mortem analysis
The 20ms pause target works with G1's concurrent marking — most GC work happens without stopping the world.
Why use AtomicReference for NodeState instead of synchronized? ▼
NodeInfo's state transitions (ALIVE → SUSPECT → DEAD) are done via CAS (Compare-And-Swap): stateRef.compareAndSet(expected, newState). If two threads try to transition a node simultaneously, only one CAS succeeds. Advantages over synchronized: no thread blocking (CAS is a single CPU instruction), no lock contention, no deadlock risk, and the JVM can optimize CAS to a hardware instruction (LOCK CMPXCHG on x86). For gossip's high-frequency state updates across many nodes, lock-free CAS scales much better than synchronized blocks.
